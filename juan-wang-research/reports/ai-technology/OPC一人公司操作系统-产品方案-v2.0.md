# 基于 OpenClaw 的 OPC 数字员工平台产品方案

> **定位**：运用 Harness Engineering 工程方法，基于 OpenClaw 构建面向 OPC（一人公司）创业者的**企业级数字员工平台**。
>
> 我们是平台方，OPC 创业者是客户。他们不需要懂 OpenClaw，只需要在平台上"雇佣"和"管理"数字员工。

---

## 第一章：产品定义

### 1.1 我们是什么

```
不是这个:                              而是这个:
"给你一套 OpenClaw，自己搭"              "给你一个平台，直接用"

类比:
OpenClaw : 我们的平台 = Linux : AWS
OpenClaw 是底层引擎                     我们是上层产品
OPC 创业者不需要知道 OpenClaw 存在        OPC 创业者只看到"数字员工"
```

### 1.2 一句话定义

> **我们是 OPC 创业者的数字员工平台——创始人登录平台，配置数字员工团队，通过飞书/钉钉指挥它们工作，平台负责一切底层运行。**

### 1.3 关键区分

| 维度 | 之前方案（偏了） | 正确方案 |
|------|----------------|---------|
| **我们的角色** | 卖工具给创始人 | **运营平台，服务创始人** |
| **用户看到的** | OpenClaw 配置文件 | **数字员工管理界面** |
| **用户需要做的** | 编辑 AGENTS.md、配 JSON | **在 Web 界面点击配置** |
| **底层引擎** | 暴露给用户 | **完全封装，用户无感** |
| **部署方式** | 用户自己搭 VPS | **我们的云 / 私有化交付** |
| **商业模式** | 开源+订阅 | **SaaS 订阅 / 私有化许可** |
| **技术门槛** | 需要懂命令行 | **零技术门槛** |

## 第二章：Harness Engineering 如何指导平台设计

### 2.1 Harness Engineering 六大组件 → 平台六大能力

Harness Engineering 的核心洞见：**Agent 不是难点，Harness 才是。** 模型能力趋同，真正拉开差距的是 Agent 运行环境的设计。

我们将 Harness Engineering 的六大组件，转化为平台的六大核心能力：

| Harness 组件 | 平台能力 | OPC 创始人看到的 | 底层 OpenClaw 实现 |
|-------------|---------|----------------|------------------|
| **工具集成层** | 系统连接 | "连接我的 CRM/邮件/支付" | MCP + Skills + Hooks |
| **记忆与状态** | 公司大脑 | "数字员工记住了所有客户信息" | MEMORY.md + LCM + Memory Search |
| **上下文工程** | 岗位说明书 | "配置数字员工的职责和知识" | Workspace 7 文件 + Skills 渐进加载 |
| **规划与分解** | 团队协作 | "CEO 员工自动分配任务" | Sub-agents + sessions_send/spawn |
| **验证与护栏** | 权限管控 | "重要操作需要我确认" | Tool Policy + Exec Approvals + Sandbox |
| **模块化扩展** | 能力市场 | "给员工添加新技能" | ClawHub Skills + MCP 连接器 |

### 2.2 Harness Engineering 的三个核心原则在平台中的体现

**原则 1：Repository-Resident Knowledge（知识驻留在系统中）**

```
传统方式: OPC 创始人的知识在大脑里 / 散落在各种文档中
         → Agent 每次对话都要重新告诉它背景
         → 知识丢失、不一致

平台方式: 知识结构化存储在平台中
         → "公司大脑"模块：产品信息、客户画像、定价策略、竞品情报
         → 所有数字员工共享同一个知识库
         → 知识持续积累，越用越聪明

底层实现: 
  MEMORY.md → 平台自动维护
  shared/knowledge/ → 结构化知识库
  Memory Search → 语义检索
```

**原则 2：Mechanical Invariants（机械式不变量）**

```
传统方式: 靠 Agent 的 prompt 约束行为
         → prompt 注入可能绕过
         → 行为不可预测

平台方式: 护栏在系统层面强制执行，不依赖 prompt
         → 权限: Tool Policy 白名单，数字员工只能用被授权的工具
         → 审批: 高风险操作强制人类确认（Exec Approvals）
         → 隔离: 每个数字员工在沙箱中运行（Sandbox）
         → 限额: 金额/频率/范围 限制在系统层面硬编码

底层实现:
  tools.policy → allow/deny 精确控制
  tools.exec.approvals → 命令级审批
  sandbox → Docker 隔离
  Safe Bins → 安全命令白名单
```

**原则 3：Progressive Disclosure（渐进式披露）**

```
传统方式: Agent 的 system prompt 塞满了所有信息
         → 上下文爆炸、成本高、质量下降

平台方式: 数字员工按需获取知识
         → 基础信息常驻（岗位说明书、核心规则）
         → 专业知识按需加载（客户详情、产品文档）
         → 技能按需激活（只在需要时加载 Skill 指令）

底层实现:
  Workspace context → 基础常驻
  Skills → 按 description 匹配后才加载 SKILL.md
  Memory Search → 语义检索按需召回
  LCM compaction → 历史压缩，保留关键
```

### 2.3 平台设计准则（从 Harness Engineering 推导）

| # | 准则 | 说明 |
|---|------|------|
| 1 | **用户配置意图，平台生成 Harness** | 创始人说"我要一个销售员工"，平台自动生成 AGENTS.md/SOUL.md/Tool Policy/Cron |
| 2 | **护栏在系统层，不在 prompt 层** | 权限、审批、隔离由平台引擎强制执行 |
| 3 | **知识结构化，自动积累** | 平台自动维护公司知识库，数字员工越用越懂业务 |
| 4 | **渐进式上下文，控制成本** | 按需加载知识和技能，不浪费 token |
| 5 | **可观测，可审计** | 每个数字员工的行为有完整日志，创始人可回溯 |
| 6 | **失败安全** | Agent 出错时回退到人类确认，不自动执行危险操作 |

### 3.2 关键设计决策

**决策 1：一个租户 = 一个 Gateway 实例**

```
为什么:
  - OpenClaw 的安全模型是"一个 Gateway = 一个信任域"
  - 租户间需要完全隔离（数据、会话、记忆、配置）
  - Gateway 轻量（Node.js 单进程，内存 ~200MB），可大规模部署

实现:
  - K8s Pod per tenant，或 Docker container per tenant
  - 平台 API 层负责创建/销毁/配置 Gateway 实例
  - 共享模型路由层做成本优化
```

**决策 2：平台生成 Harness，用户配置意图**

```
用户在 Web 管理台操作:              平台自动生成:
─────────────────────              ──────────────
创建"销售数字员工"                  → agents.list[{id:"sales"}]
填写职责描述                       → sales/AGENTS.md
选择性格"专业而不激进"              → sales/SOUL.md
勾选"可以发邮件"                   → tools.policy: allow:["message"]
勾选"报价需确认"                   → tools.exec.approvals 规则
设置"每天 14:00 检查跟进"          → cron job
连接 CRM                          → MCP 配置
上传产品手册                       → shared/knowledge/product.md
```

**决策 3：双界面——管理台（配置）+ IM 工作台（使用）**

```
Web 管理台（管理数字员工）:
  - 创建/配置/启停数字员工
  - 管理知识库
  - 连接外部系统
  - 查看运营数据和审计日志
  - 管理订阅和计费

IM 工作台（指挥数字员工工作）:
  - 在飞书/钉钉/Telegram 中与数字员工对话
  - 接收主动推送（日报/告警/审批请求）
  - 确认/驳回操作
  - 日常工作指令

为什么分开:
  - 管理是低频操作（周/月），适合 Web
  - 工作是高频操作（每天），适合 IM（创始人已经在用的工具）
  - 创始人不需要打开新 App，在现有 IM 里就能指挥数字员工
```

### 3.3 平台服务层（我们需要构建的部分）

| 模块 | 功能 | 技术选型 |
|------|------|---------|
| **租户管理** | 注册/登录、订阅、租户隔离、配额管理 | Next.js + Supabase Auth |
| **员工管理台** | 创建/配置数字员工、模板选择、参数调整 | React + REST API → 操作 OpenClaw 配置 |
| **知识库管理** | 文档上传、结构化录入、自动抽取、向量化 | 文件上传 → 写入 shared/knowledge/ + Memory |
| **连接器市场** | MCP/Skill 一键安装、OAuth 授权、连接状态 | mcporter 封装 + OAuth flow |
| **运营看板** | 工作量统计、token 用量、响应时间、异常率 | OpenClaw logs → 聚合 → Grafana/自建 |
| **审计日志** | 操作历史、决策回溯、合规报表 | Session transcripts → 结构化存储 |
| **计费引擎** | 按员工数+用量计费、订阅管理、发票 | Stripe/支付宝 + 用量追踪 |
| **Gateway 编排** | 租户 Gateway 生命周期管理 | K8s Operator / Docker API |

## 第四章：数字员工产品设计

### 4.1 "数字员工"而非 "Agent"

面向 OPC 创始人，我们不说 Agent——我们说**数字员工**。

| Agent 术语 | 数字员工术语 | 用户界面呈现 |
|-----------|------------|------------|
| Agent | 数字员工 | 头像 + 姓名 + 岗位 |
| AGENTS.md | 岗位说明书 | 表单：职责描述 |
| SOUL.md | 性格设定 | 下拉：专业/活泼/严谨/温和 |
| Tool Policy | 工作权限 | 勾选框：可以发邮件 ✅ 可以查数据 ✅ |
| Exec Approvals | 审批规则 | 配置：报价>5000 需确认 |
| Cron | 日程安排 | 日历：每天 9:00 发布内容 |
| Skills | 技能 | 技能卡片：安装/卸载 |
| MCP | 系统连接 | 连接器：一键绑定 CRM |
| Memory | 工作记忆 | 知识库：可浏览/编辑 |

**平台后台执行**（用户不可见）：

```

2. 生成 SOUL.md:
   基于"性格"选项，映射到预设的人设模板

3. 配置 Tool Policy:
   勾选的权限 → tools.policy.allow/deny 规则

4. 配置 Exec Approvals:
   审批规则 → tools.exec.approvals 配置

5. 配置 Cron Jobs:
   日程 → cron.jobs[] 条目

6. 配置 MCP:
   连接的系统 → mcporter config

7. 更新 Gateway:
   agents.list[] 添加新 Agent → hot reload
```

**关键代码**：Gateway Manager

```python
# gateway_manager.py (简化)
class GatewayManager:
    
    def create_tenant_gateway(self, tenant_id: str, config: dict):
        """为新租户创建 Gateway 实例"""
        
        # 1. 生成 workspace 目录
        workspace = f"/data/tenants/{tenant_id}/workspace"
        os.makedirs(workspace, exist_ok=True)
        
        # 2. 生成 openclaw.json
        gateway_config = self._build_config(tenant_id, config)
        write_json(f"/data/tenants/{tenant_id}/openclaw.json", gateway_config)
        
        # 3. 启动 Docker 容器
        container = docker_client.containers.run(
            image="openclaw:latest",
            name=f"gateway-{tenant_id}",
            environment={
                "OPENCLAW_CONFIG_PATH": f"/data/tenants/{tenant_id}",
                "OPENCLAW_STATE_DIR": f"/data/tenants/{tenant_id}/state",
            },
            volumes={
                f"/data/tenants/{tenant_id}": {"bind": "/root/.openclaw", "mode": "rw"}
            },
            ports={},  # 内部网络通信，不暴露端口
            network="openclaw-net",
            detach=True,
            restart_policy={"Name": "unless-stopped"}
        )
        
        return {"tenant_id": tenant_id, "container_id": container.id}
    
    def add_employee(self, tenant_id: str, employee: dict):
        """添加数字员工 = 添加 Agent + 生成 Workspace"""
        
        # 1. 生成 AGENTS.md
        agents_md = self._render_agents_md(employee)
        write_file(f".../{employee['id']}/AGENTS.md", agents_md)
        
        # 2. 生成 SOUL.md
        soul_md = self._render_soul_md(employee['personality'])
        write_file(f".../{employee['id']}/SOUL.md", soul_md)
        
        # 3. 配置 Tool Policy
        tool_policy = self._build_tool_policy(employee['permissions'])
        
        # 4. 更新 openclaw.json
        config = read_json(f"/data/tenants/{tenant_id}/openclaw.json")
        config['agents']['list'].append({
            'id': employee['id'],
            'model': employee.get('model', 'anthropic/claude-sonnet-4-5')
        })
        # ... 合并 tool policy, cron, etc.
        write_json(f"/data/tenants/{tenant_id}/openclaw.json", config)
        
        # 5. 触发 Gateway hot reload
        self._reload_gateway(tenant_id)
```

## 第八章：总结

### 8.1 核心逻辑链

```
Harness Engineering 洞见
  → Agent 不是难点，Harness 才是
  → Harness = 上下文 + 记忆 + 护栏 + 工具 + 编排 + 扩展

OpenClaw 已有完整 Harness 能力
  → Workspace 7 文件 + LCM + Tool Policy + Cron + Skills + MCP

OPC 创业者的痛点
  → 管理 AI 本身成了时间黑洞
  → 需要的不是工具，是"数字员工团队"

我们的产品
  → 基于 OpenClaw 引擎
  → 封装成 SaaS 平台
  → 用户只看到"数字员工"
  → Harness Engineering 在底层保障可靠运行
  → 6 周 MVP，验证商业模式
```

```
产品策略文档 (v1.2)
```
