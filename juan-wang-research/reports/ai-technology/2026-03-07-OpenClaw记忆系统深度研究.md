# OpenClaw 记忆系统深度研究：长期/短期记忆架构与多Agent实践

> **这不是一篇简单的工具介绍。**  
> **这是一篇关于 AI Agent 如何"记住"的深度技术分析。**

---

## 摘要

OpenClaw 的记忆系统代表了一种**返璞归真**的设计哲学：在向量数据库大行其道的时代，它选择了**纯文本 Markdown**作为记忆的基石。本文深入分析 OpenClaw 的双层记忆架构、混合检索机制、多 Agent 场景下的记忆隔离与共享，以及这一设计对 AI Agent 开发的启示。

---

## 第一部分：为什么 OpenClaw 选择文件优先？

### 1.1 设计哲学

大多数 RAG 系统依赖向量数据库作为记忆后端。OpenClaw 选择了一条不同的路：

```
传统 RAG:  文本 → 向量化 → 存入向量数据库 → 检索
OpenClaw: 文本 → Markdown 文件 → 混合检索 → 直接读取
```

**核心理念**：

| 特性 | 向量数据库 | OpenClaw 文件系统 |
|------|-----------|------------------|
| **透明度** | 黑盒，需要查询才能看 | 直接打开文件查看 |
| **可编辑性** | 需要专门工具 | 任何文本编辑器 |
| **版本控制** | 复杂 | Git 友好 |
| **可移植性** | 供应商锁定 | 纯文本，随处可用 |
| **信任度** | 不确定性 | 100% 可审计 |

### 1.2 记忆系统的价值主张

> "The most reliable memory system is also the most transparent one."
> —— Zen van Riel, OpenClaw 贡献者

文件优先的设计让用户能够：

1. **看见 AI 记住了什么** —— 打开文件就能看到
2. **修正错误** —— 直接编辑文件
3. **追溯历史** —— Git 记录每次变更
4. **迁移自如** —— 复制文件就能迁移

### 2.2 短期记忆：Daily Logs

**位置**: `memory/YYYY-MM-DD.md`

**特点**：
- 📝 **追加写入** —— 每次会话追加内容，不覆盖
- 🔄 **自动加载** —— 会话开始时加载今天 + 昨天
- ⏰ **时效性** —— 短期上下文，可能随时间过时
- 🧹 **自动压缩前刷新** —— 上下文窗口满时自动保存

**示例**：

```markdown
# 2026-03-07

## 用户偏好更新
- 用户表示更喜欢简洁的回复风格
- 决定使用 TypeScript 而不是 JavaScript

## 任务记录
- 完成了 API 网关的重构
- 修复了用户认证的 bug

## 待办
- 明天需要检查数据库迁移脚本
```

### 2.3 长期记忆：MEMORY.md

**位置**: `MEMORY.md`

**特点**：
- 🧠 **精心策展** —— 从 Daily Logs 提炼的关键知识
- 🔒 **隐私保护** —— 仅在私密会话加载，不在群组/频道暴露
- 📊 **结构化** —— 按主题组织，易于检索
- 🔄 **持续更新** —— AI 定期审查并更新

**示例**：

```markdown

- 名字: 张三
- 角色: 全栈开发者
```

## 第三部分：混合检索机制

### 3.1 为什么需要混合检索？

纯向量检索有局限性：

| 场景 | 向量检索 | BM25 检索 |
|------|----------|-----------|
| "gateway host" ≈ "运行网关的机器" | ✅ 语义匹配 | ❌ 无关键词匹配 |
| "ERR_CONNECTION_REFUSED" | ❌ 无语义相似 | ✅ 精确匹配 |
| "handleUserAuth()" | ❌ 函数名无语义 | ✅ 精确匹配 |
| "用户偏好是什么" | ✅ 理解意图 | ❌ 关键词弱 |

### 3.2 混合检索算法

```typescript
// 权重融合算法
function mergeHybridResults(params: {
  vector: HybridVectorResult[];
  keyword: HybridKeywordResult[];
  vectorWeight: number;  // 默认 0.7
  textWeight: number;    // 默认 0.3
}): Array<...> {
  
  // 按 chunk ID 合并
  const byId = new Map<string, {...}>();
  
  for (const r of params.vector) {
    byId.set(r.id, { ...r, vectorScore: r.vectorScore, textScore: 0 });
  }
  
  for (const r of params.keyword) {
    const existing = byId.get(r.id);
    if (existing) {
      existing.textScore = r.textScore;
    }
  }
  
  // 加权评分
  const merged = Array.from(byId.values()).map((entry) => {
    const score = params.vectorWeight * entry.vectorScore
                + params.textWeight * entry.textScore;
    return { ...entry, score };
  });
  
  return merged.sort((a, b) => b.score - a.score);
}
```

### 4.4 自动记忆刷新

**触发条件**：上下文接近压缩阈值

**机制**：
```json5
{
  compaction: {
    reserveTokensFloor: 20000,
    memoryFlush: {
      enabled: true,
      softThresholdTokens: 4000,
      systemPrompt: "Session nearing compaction. Store durable memories now.",
      prompt: "Write any lasting notes to memory/YYYY-MM-DD.md; reply with NO_REPLY if nothing to store."
    }
  }
}
```

**效果**：
- 重要信息不会在上下文压缩时丢失
- 自动保存到 Daily Logs
- 用户无感知（NO_REPLY）

### 5.2 记忆隔离模型

**每个 Agent 独立拥有**：

| 资源 | 说明 |
|------|------|
| `workspace/` | 独立工作区 |
| `MEMORY.md` | 独立长期记忆 |
| `memory/` | 独立短期记忆 |
| `sessions/` | 独立会话历史 |
| `auth profiles` | 独立认证配置 |
| `tool allow/deny` | 独立工具权限 |

**设计原理**：
- 安全隔离：一个 Agent 的错误不影响其他
- 记忆隔离：避免上下文污染
- 权限隔离：不同 Agent 不同能力

### 5.3 协调者-专家模式

**最可靠的多 Agent 模式**：

```
┌─────────────────────────────────────────────────────────────┐
│                    Coordinator-Specialist 模式               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │                  Coordinator                         │   │
│   │  • 接收任务                                           │   │
│   │  • 分类和分解                                         │   │
│   │  • 调用 sessions_spawn 委派                           │   │
│   │  • 拥有 MEMORY.md                                     │   │
│   │  • 汇总结果并持久化                                    │   │
│   └────────────────────────┬────────────────────────────┘   │
│                            │                                │
│            ┌───────────────┼───────────────┐                │
│            ▼               ▼               ▼                │
│   ┌────────────┐  ┌────────────┐  ┌────────────┐          │
│   │  Research  │  │   Coding   │  │   Review   │          │
│   │ Specialist │  │ Specialist │  │ Specialist │          │
│   │            │  │            │  │            │          │
│   │ 无记忆     │  │ 无记忆     │  │ 无记忆     │          │
│   │ 完成即止   │  │ 完成即止   │  │ 完成即止   │          │
│   └────────────┘  └────────────┘  └────────────┘          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**配置示例**：

```json5
// Coordinator 配置
{
  id: "coordinator",
  systemPrompt: `
    你分解传入任务并通过 sessions_send 委派给专家代理。
    你拥有 MEMORY.md。
    汇总结果后再存储。
    永不递归——如果专家把任务返回给你，汇总并关闭。
  `,
  tools: ["sessions_send", "sessions_list", "memory_search", "read", "write"]
}

// Specialist 配置
{
  id: "research-specialist",
  systemPrompt: `
    你处理协调者委派的研究任务。
    完成任务，返回简洁摘要，然后停止。
  `,
  tools: ["web_search", "read", "write"],
  memory: { enabled: false }  // 关键：无状态
}
```

**关键约束**：
- 专家 Agent 应该**无状态** (`memory.enabled: false`)
- 协调者**从不接受**从专家返回的任务
- 这防止无限递归循环

### 5.4 共享状态模式

**文件为基础的协调**：

```
共享工作区/
├── goal.md          # 当前任务和分解
├── plan.md          # 执行计划和子任务分配
├── status.md        # 子任务状态
└── log.md           # 追加式执行日志
```

**工作流**：
1. 协调者写入 `goal.md` 和 `plan.md`
2. 专家读取并更新 `status.md`
3. 所有专家追加写入 `log.md`
4. 协调者监控并聚合

**优点**：
- 完全可审计
- 无需复杂协调协议
- 易于调试

**缺点**：
- 文件 I/O 延迟
- 需要处理并发写入

### 5.5 子代理深度限制

| 深度 | Session Key | 角色 | 能否派生子代理 |
|------|-------------|------|---------------|
| 0 | `agent:<id>:main` | 主代理 | ✅ 始终可以 |
| 1 | `agent:<id>:subagent:<uuid>` | 子代理 | ⚠️ 仅当 maxSpawnDepth >= 2 |
| 2 | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | 子子代理 | ❌ 永远不能 |

**配置**：
```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2,        // 允许两层嵌套
        maxChildrenPerAgent: 5,  // 每个代理最多5个活跃子代理
        maxConcurrent: 8,        // 全局并发上限
        runTimeoutSeconds: 900   // 超时设置
      }
    }
  }
}
```

### 5.6 防止常见失败模式

#### 无限递归循环

**症状**：协调者 → 专家 A → 协调者 → 专家 A → ...

**解决**：
```json5
// 在协调者提示中强制
"永不接受从专家返回的任务——汇总并关闭"

// 或在工具配置中阻止
{
  tools: {
    deny: ["sessions_send"]  // 专家不能调用 sessions_send 到协调者
  }
}
```

#### 共享状态死锁

**症状**：专家 A 等待专家 B，专家 B 等待专家 A

**解决**：
- 为每个任务分配唯一 ID
- 检查状态后再认领
- 实现超时机制
- 使用 `teammate_shutdown` 终止卡住的代理

#### 成本失控

**症状**：每个传递都增加 token 开销

**解决**：
- 专家应返回结构化输出，而非长篇总结
- 协调者应合并相似任务
- 设置每代理 token 限制
- 监控 token 使用

## 第六部分：最佳实践

### 6.1 记忆写入原则

**用户应该明确要求写入**：

```
❌ AI 自己判断重要性（可能遗漏）
✅ 用户说 "记住这个偏好"
✅ 用户说 "把这个写入 MEMORY.md"
```

**AI 应该主动询问**：

```
"我需要记住这个决策吗？"
"这是一个持久偏好还是临时选择？"
```

### 6.2 Daily Logs vs MEMORY.md

| 内容类型 | 存储位置 | 示例 |
|----------|----------|------|
| 临时决策 | Daily Log | "今天用 Docker 调试" |

| 持久偏好 | MEMORY.md | "用户偏好深色主题" |
| 重要决策 | MEMORY.md | "选择 PostgreSQL 而非 MySQL" |
| 项目约定 | MEMORY.md | "测试覆盖率 > 80%" |

## 第八部分：总结与建议

### 8.1 核心设计洞察

| 洞察 | 说明 |
|------|------|
| **文件即真理** | AI 只记住写入磁盘的内容 |
| **双层分离** | 临时日志 vs 精选知识 |
| **隐私优先** | MEMORY.md 仅私密加载 |
| **混合检索** | 向量 + BM25 各有所长 |
| **多 Agent 隔离** | 独立工作区、独立记忆 |

### 8.2 实践建议

| 场景 | 建议 |
|------|------|
| **个人助手** | 使用默认配置，定期审查 MEMORY.md |
| **团队协作** | 共享工作区 + 协调者模式 |
| **专家系统** | 专家无状态，协调者持记忆 |
| **安全敏感** | 严格区分私密/群组会话 |
| **长期运行** | 配置记忆刷新，监控 token 消耗 |

### 8.3 未来方向

- **Teams RFC**：共享任务列表 + 代理邮箱
- **文件锁**：原子写入支持
- **记忆压缩**：智能摘要 Daily Logs
- **跨代理同步**：实时知识共享

### 完整记忆配置

```json5
{
  agents: {
    defaults: {
      workspace: "~/.openclaw/workspace",
      
      // 记忆配置
      memorySearch: {
        provider: "auto",  // auto | local | openai | gemini | voyage | mistral | ollama
        model: "text-embedding-3-small",
        extraPaths: ["../team-docs"],
        
        // QMD 后端（实验性）
        backend: "sqlite",  // sqlite | qmd
        citations: "auto",  // auto | on | off
        
        // 检索限制
        maxResults: 10,
        maxSnippetChars: 1000
      },
      
      // 自动记忆刷新
      compaction: {
        reserveTokensFloor: 20000,
        memoryFlush: {
          enabled: true,
          softThresholdTokens: 4000
        }
      },
      
      // 子代理配置
      subagents: {
        maxSpawnDepth: 2,
        maxChildrenPerAgent: 5,
        maxConcurrent: 8,
        runTimeoutSeconds: 900
      }
    }
  }
}
```
