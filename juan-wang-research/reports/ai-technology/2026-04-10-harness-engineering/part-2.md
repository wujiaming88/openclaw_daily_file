# Part 2：Anthropic —— "Harness Design for Long-Running Application Development"

> 作者：Prithvi Rajasekaran (Labs 团队) | 发布：2026-03-24 | 来源：anthropic.com/engineering

## 核心问题

Prithvi 聚焦两个交叉问题：
1. 让 Claude 生成**高质量前端设计**
2. 让 Claude **无人干预地构建完整应用**

通过 prompt engineering 和 harness 设计都提升了性能，但最终**都撞到天花板**。

## 两个关键失败模式

### 失败模式 1：上下文焦虑（Context Anxiety）

> 模型在上下文窗口接近填满时，开始**过早收尾工作**。

- Sonnet 4.5 表现尤为严重
- **Compaction（压缩）不够**——虽保留连续性，但不给 Agent 全新起点，焦虑仍在
- **Context Reset（上下文重置）**才是解法：完全清空 → 新 Agent + 结构化交接产物
- Opus 4.5 基本消除了这个问题 → 可以去掉重置，用 SDK 自动压缩

### 失败模式 2：自我评估偏差

> Agent 评估自己的工作时，**总是自信地给出好评**——即使质量明显平庸。

- 设计任务尤其严重（无法像测试那样二元验证）
- 解法：**分离生成器和评估器** —— 灵感来自 GAN

## 突破：GAN 启发的 Generator-Evaluator 架构

### 前端设计实验

**核心洞察**：
1. 美学不能简化为分数，但可以用**编码了设计原则的评分标准**来改进
2. 分离生成和评判 → 创建反馈循环

**四个评分维度**：
| 维度 | 权重 | 说明 |
|------|------|------|
| **Design Quality** | 高 | 整体感觉是否统一？色彩/排版/布局是否形成独特氛围？ |
| **Originality** | 高 | 是否有自定义决策？AI slop 直接扣分（紫色渐变+白卡片） |
| **Craft** | 低 | 技术执行力（排版层次、间距、对比度） |
| **Functionality** | 低 | 可用性（用户能理解界面并完成任务吗？） |

- 重点惩罚"AI slop"，鼓励审美冒险
- 评估器用 **Playwright MCP** 实际操作页面（不是看截图）
- **5-15 轮迭代**，每轮策略：得分趋势好 → 细化；不行 → 整体转向
- 完整运行**最长 4 小时**

**惊人案例**：荷兰艺术博物馆网站
- 前 9 轮迭代：一个干净的深色主题着陆页
- 第 10 轮：**彻底推翻**，重新想象为 3D 空间体验——CSS 透视的棋盘地板、墙上自由悬挂的画作、门廊导航替代滚动
- **"这是我从未见过的单次生成创造性飞跃。"**

### 全栈编码 —— 三 Agent 架构

```
┌────────────────────────────────────┐
│           Planner                   │
│  1-4 句提示 → 完整产品规格           │
│  "要有野心，不要过度指定技术细节"     │
│  主动编织 AI 功能                    │
└──────────┬─────────────────────────┘
           ↓ 产品规格
┌────────────────────────────────────┐
│          Generator                  │
│  按 Sprint 逐个实现功能             │
│  React + Vite + FastAPI + SQLite   │
│  Sprint 前与 Evaluator 协商合同     │
└──────────┬─────────────────────────┘
           ↓ 实现 + 自评
┌────────────────────────────────────┐
│          Evaluator                  │
│  Playwright MCP 实际操作应用         │
│  按合同逐条验证                     │
│  任何维度低于阈值 → Sprint 失败      │
│  详细反馈 → 回到 Generator          │
└────────────────────────────────────┘
```

### Sprint 合同机制（关键创新）

每个 Sprint 开始前：
1. Generator **提议**要构建什么 + 如何验证成功
2. Evaluator **审查**提议是否合理
3. 双方**协商**直到达成一致
4. 然后 Generator 执行 → Evaluator QA

**通信方式**：通过文件——一个 Agent 写文件，另一个读取并回复。

### 对比实验

| 指标 | Solo Agent | 三 Agent Harness |
|------|-----------|-----------------|
| 时间 | 20 分钟 | 6 小时 |
| 成本 | $9 | $200 |
| 功能 | 基础编辑器+坏掉的游戏 | 16 功能、10 Sprint、含 AI 功能 |
| 游戏可玩 | ❌ 不响应输入 | ✅ 可移动、有物理（有瑕疵） |
| 设计质量 | 空间浪费、UI 死板 | 全视口、一致视觉语言 |

## Opus 4.6 的影响

作者后来用 Opus 4.6 复现实验：
- **不再需要 Evaluator** —— 单 Agent 的输出质量接近三 Agent 系统
- 游戏可玩、UI 精致、甚至加入了之前不存在的功能
- **"Opus 4.6 的独立能力足够强，以至于它不需要外部评估器来保持正轨。"**

> 核心问题变成：**当模型足够好时，harness 应该如何演化？**
