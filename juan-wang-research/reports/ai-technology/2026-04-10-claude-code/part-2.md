## QueryEngine —— 46K 行的核心大脑

QueryEngine.ts 是 Claude Code 最核心的单文件模块，负责：

1. **所有 LLM API 调用**：发送请求、接收流式响应
2. **上下文预算管理**：实时追踪 token 使用，触发压缩
3. **自愈查询循环**：上下文接近上限时自动触发压缩，不崩溃不中断
4. **缓存策略**：Anthropic prompt caching 集成
5. **重试与限速**：rate limit 处理、自动重试
6. **流式处理**：token-by-token 流式输出

**为什么放在一个文件？** 所有触碰模型 API 的逻辑集中管理——重试逻辑、限速处理、上下文预算、流式错误可以统一推理，避免分散后的不一致。

## Agentic Loop（Agent 循环）

Claude Code 的工作流程分三个阶段，循环执行：

```
┌─────────────────────────────┐
│  1. 收集上下文（Gather）      │
│  搜索文件、读代码、理解结构   │
└──────────┬──────────────────┘
           ↓
┌─────────────────────────────┐
│  2. 执行行动（Act）           │
│  编辑文件、运行命令、调 API   │
└──────────┬──────────────────┘
           ↓
┌─────────────────────────────┐
│  3. 验证结果（Verify）        │
│  运行测试、检查输出、确认修复 │
└──────────┬──────────────────┘
           ↓
       循环直到完成
       （用户可随时中断）
```

- **每个工具调用的结果反馈到下一步决策**
- **用户始终在循环中**：可中断、重定向、补充上下文
- 简单问题可能只需 Gather，复杂 bug 修复需要反复循环

## 工具系统 —— 40+ 独立门控模块

### 工具分类

| 类别 | 能力 | 示例工具 |
|------|------|---------|
| 文件操作 | 读写、创建、重命名 | Read, Write, Edit |
| 搜索 | 文件名/内容搜索 | Grep, Glob, Find |
| 执行 | Shell 命令、构建、测试 | BashTool |
| Web | 搜索、获取文档 | WebSearch, WebFetch |
| 代码智能 | 类型错误、跳转定义 | LSP 集成 |
| 编排 | 子 Agent、提问 | AgentTool, Clarify |

### 关键设计原则

1. **每个工具独立定义 schema、权限、执行逻辑** —— 无共享可变状态
2. **BashTool vs FileReadTool 权限完全独立** —— Bash 可改系统状态，Read 只读
3. **AgentTool 让子 Agent 成为一级工具** —— 扁平化架构，不需要特殊编排层
4. **基础工具定义 ~29,000 行 TypeScript** —— 严格 schema 验证 + 权限执行 + 错误处理

## 多 Agent 编排 —— Coordinator/Worker/Mailbox 模式

### 架构

```
┌─────────────────────────┐
│      Coordinator         │
│  (协调器/主 Agent)        │
│  - 分配子任务             │
│  - 审批危险操作           │
│  - 合并结果               │
└──────┬──────────────────┘
       ↓ 分配任务
┌──────┬──────┬───────────┐
│Worker│Worker│Worker     │
│  A   │  B   │  C        │
│      │      │           │
│  ┌───┴───┐  │           │
│  │Mailbox│  │  (共享内存) │
│  └───────┘  │           │
└─────────────┴───────────┘
```

### Mailbox 模式（关键创新）

- Worker 执行到危险操作时 → 不自主审批
- 发送请求到 Coordinator 的 Mailbox → 等待
- Coordinator 评估后 approve/reject
- **原子认领机制**：防止两个 Worker 同时处理同一审批
- **共享内存空间**：所有 Agent 维持一致上下文

### 设计意义

与"完全自治"的多 Agent 设计形成鲜明对比：
- 并行执行 + 集中审批 = **有控制的并行性**
- 模型无法通过说服绕过安全检查 = 安全机制与推理分离
